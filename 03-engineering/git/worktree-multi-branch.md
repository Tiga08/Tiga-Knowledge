# Git Worktree 多分支并行开发

> **环境**：Git
> **最后更新**：2026-03-30

## 概述

`git worktree` 允许在同一个仓库中同时开启多个工作目录，每个目录对应不同的分支，无需频繁使用 `git stash` 或 `git commit` 来切换上下文。

---

## 1. 核心概念
传统的 `git clone` 会创建一个包含 `.git` 文件夹的工作目录。而 `git worktree` 允许你将一个仓库“拆分”到多个文件夹中，这些文件夹共享同一个 `.git` 数据库。

**主要优势：**
* **并行开发**：可以在一个目录跑长耗时的测试，在另一个目录修复紧急 Bug。
* **无需暂存**：切换分支时不需要 `stash` 还没写完的代码。
* **极速切换**：比重新 `clone` 仓库快得多，且节省磁盘空间。

---

## 2. 常用命令指南

### ❶ 添加工作树 (`add`)
这是最常用的命令，用于创建一个新的目录并检出分支。

```bash
# 语法：git worktree add <路径> <分支名>
git worktree add ../my-feature-branch feature-alpha
```
* **解释**：在当前目录的上一级创建一个名为 `my-feature-branch` 的文件夹，并检出 `feature-alpha` 分支。
* **创建新分支并关联**：
    ```bash
    git worktree add -b new-hotfix ../hotfix-dir main
    ```

### ❷ 查看工作树列表 (`list`)
查看当前仓库关联的所有工作树及其所在路径和分支。

```bash
git worktree list
```
> **提示**：输出结果的第一行通常是你的“主工作区”。

### ❸ 移除工作树 (`remove`)
当你完成了某个分支的工作，需要清理该目录。

```bash
git worktree remove <路径或名称>
```
* **注意**：如果工作区有未提交的改动，需要加上 `--force` 强制删除。

### ❹ 清理过期信息 (`prune`)
如果你手动删除了工作树的文件夹（而不是通过 `remove` 命令），Git 的记录里可能还会残留信息。

```bash
git worktree prune
```

## 3. 实际应用场景（最佳实践）

### 场景 A：紧急修复 Bug
你在 `main` 分支开发大功能，突然需要修复一个 Bug。
1.  执行 `git worktree add ../temp-fix main`。
2.  进入 `../temp-fix` 目录修复并提交。
3.  回到主目录继续开发，完全不需要动主目录的改动。

### 场景 B：对比两个分支的代码
想同时打开两个 VS Code 窗口对比 `v1.0` 和 `v2.0` 的代码实现。
1.  执行 `git worktree add ../v1-code v1.0`。
2.  现在你可以同时在两个目录下查看/运行代码。

---

## 4. 注意事项（避坑指南）
* **禁止双重检出**：Git 不允许两个工作树同时检出同一个分支（防止指针冲突）。如果你需要这么做，必须其中一个处于“游离头指针”状态。
* **相对路径**：建议将工作树放在主仓库目录的**同级**位置，而不是主仓库内部，以避免 Git 扫描嵌套目录导致性能下降。
* **主仓库不能动**：不要直接删除最初 `clone` 下来的那个主目录，否则其他 worktree 都会因为找不到 `.git` 物理路径而失效。
